修复构建错误，确保应用能在本地完整构建并运行，建立可持续的类型管理机制，并完成覆盖全部功能的 E2E 测试验证。

## 背景
项目已完成 22 个功能模块开发，但构建阶段出现 TypeScript 类型错误，导致 `wails build` 失败。根本原因：前端代码直接依赖 Wails 生成绑定的内部路径，当绑定生成结构变化时出现类型不匹配。此外，项目缺乏端到端的功能验证，无法确保各模块在集成后正常工作。

## 风险与注意事项
1. Wails 绑定重新生成可能覆盖 wailsjs 目录，但类型适配层可隔离此影响
2. E2E 测试涉及桌面 GUI 操作，可能需要特殊测试框架或手动验证辅助
3. 确保修改后 `frontend/dist` 内容正确更新
4. macOS 签名问题可能在本地构建时出现，但不影响功能验证
5. AI 模块 E2E 测试需要配置 API Key，测试时应使用 Mock 或测试环境
6. LSP 模块 E2E 测试需要安装对应语言服务器（如 gopls）

## E2E 测试策略说明

考虑到 AI IDE 是基于 Wails 的桌面应用，E2E 测试采用以下策略：

1. **后端 E2E**：使用 Go 测试直接调用 Wails 绑定的服务方法（FileService, GitService, TerminalService 等），验证业务逻辑正确性
2. **前端集成测试**：使用 React Testing Library 测试组件交互（适合 Git 面板、设置面板等纯 UI 组件）
3. **手动验证清单**：对于需要完整桌面环境的场景（如 Monaco 编辑器、xterm.js 终端），提供详细的手动验证步骤清单
4. **构建验证**：将 `wails build` 成功作为最重要的 E2E 指标——如果应用能完整构建，说明前后端集成基本正常

这种分层策略既保证了自动化覆盖率，又避免了过度复杂的桌面 GUI 自动化测试 setup。
